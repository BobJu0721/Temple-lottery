## 四、實體定義（Entity Definitions）

每個實體需包含以下資訊：

### [實體名稱]（Entity Name）

**說明**：這個實體代表什麼業務概念？

**資料表 / Collection 名稱**：`table_name`

#### 欄位定義

| 欄位名稱      | 資料型別     | 必填 | 預設值 | 說明                         |
| ------------- | ------------ | ---- | ------ | ---------------------------- |
| `id`          | UUID / BIGINT | ✅  | 自動產生 | 主鍵，唯一識別碼             |
| `created_at`  | TIMESTAMP    | ✅   | NOW()  | 建立時間                     |
| `updated_at`  | TIMESTAMP    | ✅   | NOW()  | 最後更新時間                 |
| `deleted_at`  | TIMESTAMP    | ❌   | NULL   | 軟刪除時間，NULL 表示未刪除  |
| `name`        | VARCHAR(255) | ✅   | —      | 名稱                         |
| `status`      | ENUM         | ✅   | active | 狀態：active / inactive      |

#### 限制與規則（Constraints & Rules）

- 主鍵（Primary Key）：`id`
- 唯一鍵（Unique Key）：`email`
- 外來鍵（Foreign Key）：`user_id` → `users.id`
- 索引（Index）：`created_at`（查詢效能），`status`（過濾常用欄位）
- 檢查約束（Check Constraint）：`age >= 0`

#### 業務規則（Business Rules）

- 說明此實體的業務邏輯規則
- 例如：「用戶刪除後訂單資料應保留 7 年」

## 五、關聯定義（Relationship Definitions）

詳細說明實體之間的關聯方式。

| 關聯             | 類型   | 外來鍵欄位         | 說明                               |
| ---------------- | ------ | ------------------ | ---------------------------------- |
| User → Order     | 1 對多 | `orders.user_id`   | 一個用戶可有多筆訂單               |
| Order → Product  | 多對多 | `order_items` 中介表 | 一筆訂單可包含多個商品，反之亦然 |

## 七、索引策略（Index Strategy）

說明為什麼在哪些欄位建立索引，並評估效能影響。

| 資料表   | 索引欄位              | 索引類型 | 建立原因                       |
| -------- | --------------------- | -------- | ------------------------------ |
| `users`  | `email`               | UNIQUE   | 登入查詢、防止重複註冊         |
| `orders` | `user_id`             | INDEX    | 查詢特定用戶所有訂單           |
| `orders` | `created_at`          | INDEX    | 依時間排序查詢                 |
| `orders` | `(user_id, status)`   | 複合索引 | 查詢特定用戶的特定狀態訂單     |

> **注意**：索引會加快查詢但降低寫入效能，需依實際查詢模式決定是否建立。

## 八、資料生命週期（Data Lifecycle）

說明資料從產生到刪除的完整週期。

- **建立（Create）**：在什麼情境下產生這筆資料？
- **更新（Update）**：哪些欄位會被更新？觸發條件為何？
- **刪除策略**：
  - **硬刪除（Hard Delete）**：直接從資料庫移除
  - **軟刪除（Soft Delete）**：標記 `deleted_at`，資料保留
- **封存（Archive）**：超過多久的資料移至冷儲存？
- **保留政策（Retention Policy）**：法規要求保留多少年？

## 九、資料遷移計畫（Migration Plan）

若為改動現有 Schema，需說明：

- **變更類型**：新增欄位 / 修改型別 / 刪除欄位 / 新增資料表
- **向下相容性**：是否 Breaking Change？舊資料如何處理？
- **遷移腳本**：附上 SQL Migration 或 ORM Migration 檔案
- **回滾方式**：若遷移失敗如何還原？

#### 遷移腳本範例格式

```sql
-- Migration: add_phone_to_users
-- Version: 20240422001
-- Author:

CREATE INDEX idx_users_phone ON users(phone);
```

## 撰寫資料模型文件的核心原則

1. **命名一致性**：欄位命名遵循統一規範（snake_case / camelCase），全專案保持一致
2. **說明業務意義**：每個欄位不只說明型別，要說明「代表什麼業務概念」
3. **明確列舉值**：所有狀態、類型欄位都要集中定義合法值，避免魔術字串
4. **索引要有理由**：每個索引都應說明為何存在，避免過度索引
5. **考慮資料成長**：預估資料量級，提前規劃分片、分區或封存策略
6. **隱私優先**：設計初期就標記 PII 欄位，確保符合隱私法規
