---
name: prd
description: 產品需求文件（Product Requirements Document，PRD）是產品開發過程中最核心的文件之一。它清楚定義「要做什麼」與「為什麼要做」，讓工程、設計、商業等團隊對齊共識，避免認知落差。

---

## 一、文件基本資訊

| 欄位     | 說明                          |
| -------- | ----------------------------- |
| 文件標題 | 產品或功能的名稱              |
| 版本號   | v1.0、v1.1…                   |
| 作者     | 負責撰寫的 PM                 |
| 審閱者   | 相關利害關係人                |
| 建立日期 | 文件創建時間                  |
| 最後更新 | 最近修改時間                  |
| 狀態     | 草稿 / 審閱中 / 已核准 / 封存 |
| 關聯文件 | 對應 ADR、設計稿、Ticket 連結 |

---

## 二、背景與目標（Background & Goals）

說明「為什麼要做這件事」。

- **問題陳述**：現在遇到什麼問題？用戶的痛點是什麼？
- **商業目標**：這個功能或產品能帶來什麼商業價值？
- **成功指標（KPI）**：如何衡量成功？
  例如：DAU 成長 20%、購買轉換率提升 5%、NPS 提高 10 分

---

## 三、目標使用者（Target Users）

明確說明「為誰而做」。

- **用戶畫像（Persona）**：描述典型使用者的特徵、行為與需求
- **用戶旅程（User Journey）**：從觸及到完成任務的完整路徑
- **主要使用場景**：在什麼情境下會使用這個功能？

---

## 四、範圍（Scope）

清楚劃定本次版本的邊界，避免需求蔓延（Scope Creep）。

- **In Scope（本次要做）**：列出這次版本包含的功能
- **Out of Scope（本次不做）**：明確說明排除的項目，並可附上未來規劃說明

---

## 五、功能需求（Functional Requirements）

描述「系統要具備什麼能力」。

### 優先級標準

| 優先級 | 定義                             |
| ------ | -------------------------------- |
| P0     | 必須有，上線阻斷條件             |
| P1     | 應該有，影響核心體驗             |
| P2     | 可以有，優化或加分項目           |

### 使用者故事格式

> 身為 **[用戶角色]**，我想要 **[完成某件事]**，以便 **[達成某個目標]**。

### 撰寫建議

- 依優先級排序列出所有功能
- 每個功能說明「做什麼」，不規定「怎麼做」（實作細節留給工程師）
- 附上對應的驗收條件（Acceptance Criteria）

---

## 六、非功能需求（Non-Functional Requirements）

描述系統品質標準，不是「功能」而是「特性」。

- **效能**：頁面載入時間 < 2 秒、API 回應時間 < 500ms
- **安全性**：資料加密規範、身份驗證、權限控制
- **可用性（SLA）**：服務可用率 99.9% 以上
- **擴展性**：支撐用戶量 10 倍成長的系統策略
- **無障礙性**：是否需符合 WCAG 2.1 AA 標準

---

## 七、UX / UI 設計

連結設計相關資產，協助團隊理解操作邏輯與視覺方向。

- **用戶流程圖（User Flow）**：描述用戶操作步驟與分支判斷
- **線框圖 / Mockup**：附上 Figma 或設計稿連結
- **互動邏輯說明**：hover、error state、empty state 等邊界情境

---

## 八、數據需求（Data Requirements）

確保產品可量測、可追蹤。

- 需蒐集哪些用戶行為數據？
- 數據埋點（Event Tracking）規格（事件名稱、觸發時機、帶入參數）
- 是否涉及個人資料？需符合哪些隱私法規（如 GDPR、個資法）？

---

## 九、相依關係（Dependencies）

列出所有外部依賴，方便風險評估與排程。

- 依賴的第三方 API 或服務
- 依賴的其他內部系統或功能模組
- 依賴的其他團隊（如法務審查、客服系統調整）

---

## 十、風險與假設（Risks & Assumptions）

提前識別不確定性，幫助團隊做好預案。

- **假設**：撰寫文件時的前提（若假設錯誤，需求可能改變）
- **風險**：可能造成延誤或失敗的因素，及對應的緩解策略

## 撰寫 PRD 的核心原則

1. **以用戶為中心**：所有需求應回扣用戶問題與商業目標
2. **具體可驗收**：每個需求應有明確的完成標準（Acceptance Criteria）
3. **避免實作細節**：PRD 說「要什麼」，不說「怎麼做」
4. **保持更新**：PRD 是活文件，需隨決策演進而更新，並記錄版本歷程
5. **提前對齊**：在撰寫完成後，安排 kick-off 會議與相關團隊確認共識
